# 命令書フォーマット

サブエージェントへの命令は以下のフォーマットで出すこと。
工程ごとの詳細度が異なる。下記の使い分けルールを厳守せよ。

## 命令書作成前の必須確認事項

命令書を作成する前に、以下を工程別に必ず確認してから記載せよ。確認なしに命令書を作成することを禁止する。

| 工程 | 確認すべき内容 | 確認方法 |
|------|---------------|---------|
| 要件定義 | 将軍閣下の命令内容を漏れなく把握しているか | 命令内容を再読する |
| データモデリング | 要件定義書に記載された全テーブルを把握しているか | `artifacts/requirements/` を Read |
| 基本設計 | 既存APIカタログ・プロジェクトルールを把握しているか | `artifacts/design/base/api-catalog.md` と `rules/project-rule.md` を Read |
| **コーディング（並列）** | **設計書に記載された全ファイルの完全パスを把握しているか** | **`artifacts/design/` 配下の設計書を Read し、1.3節（ディレクトリ構成）の完全パスを確認する** |
| **単体テスト（並列）** | **実装済みファイルの実際のパスを把握しているか** | **`artifacts/code/` 配下を確認し、実際に存在するパスを使用する** |
| ソースレビュー | 新規作成・変更された全ファイルのパスを把握しているか | コーディング報告書のファイル一覧を確認 |
| 結合テスト | 実装済みのAPI・モジュール構成を把握しているか | 設計書とコーディング報告書を確認 |

### コーディング並列命令における【担当範囲】の記載ルール

**【担当範囲】に記載するパスは、必ず設計書の1.3節（ディレクトリ構成）に記載された完全パスを使用すること。**

- **禁止:** 設計書に存在しないサブディレクトリを隊長が独自に作成・命名すること
  - 悪い例: `controller/own/` （設計書に記載がない隊長発明のサブディレクトリ）
  - 悪い例: `service/research/` （同上）
- **必須:** 設計書の記載通りのパスで分割すること
  - 良い例: 設計書が `controller/FooController.java` と記載していれば `controller/` を担当範囲とする
- **設計書にパスの記載がない場合:** コーディング命令を単一エージェントで発令するか、設計工程に差し戻して完全パスを記載させること

## 使い分けルール

| 工程 | 命令書の詳細度 | 理由 |
|------|---------------|------|
| 要件定義 | **詳細記載** | 将軍閣下の口頭指示が唯一のインプットであり、命令書に詳細を書かなければ兵が作業できない |
| データモデリング以降 | **要約レベル** | 前工程の成果物（inputDirs）に詳細が記載済みのため、命令書で繰り返す必要はない |

```
■ 完了条件
- {何をもって任務完了とするか}
═══════════════════════════════════════
```

## データモデリング以降の命令書（要約版）

前工程の成果物に詳細が記載済みのため、命令書では任務の要約と補足事項のみ記載する。
inputDirs は `pipeline/pipeline-status.json` の各工程定義を参照せよ。

```
═══════════════════════════════════════
【命令書】
発令者: 隊長 全体管理
受令者: {階級} {役職名}
発令日時: {YYYY-MM-DD HH:MM}
───────────────────────────────────────
■ 任務概要
{一文で任務を要約}

■ 参照ドキュメント
以下のディレクトリ配下の全ファイルを読み込み、内容を把握した上で作業せよ。
- {inputDir1}
- {inputDir2}

■ 補足・特記事項
- {ドキュメントに記載されていない追加指示、将軍閣下からの特別な要求、手戻り時の修正指示など。なければ「特になし」}
- 作業中に「このパイプラインにスキルがあれば効率化できる」と感じた場合は、報告書の「スキル要望」セクションに記載せよ。

■ 成果物
- {期待する成果物とその格納先}

■ 完了条件
- {何をもって任務完了とするか}
═══════════════════════════════════════
```

## サブタスク命令書（工程内並列用 — コーディング・単体テスト・強化テスト・完全テスト工程）

コーディング・単体テスト・強化テスト・完全テスト工程でタスク分割して並列実行する場合、各エージェントへの命令書に**必ず**以下のマーカーを含めること。フックがこのマーカーを解析してスコープ重複を検知する。

```
═══════════════════════════════════════
【命令書】
発令者: 隊長 全体管理
受令者: {階級} {役職名}
発令日時: {YYYY-MM-DD HH:MM}
【サブタスク: {一意なID — 例: coding-user, test-product}】
【担当範囲: {担当するディレクトリ/パッケージ — カンマ区切り}】
───────────────────────────────────────
■ 任務概要
{一文で任務を要約}

■ 参照ドキュメント
以下のディレクトリ配下の全ファイルを読み込み、内容を把握した上で作業せよ。
- {inputDir1}
- {inputDir2}

■ 担当範囲（厳守）
お前の担当範囲は以下のみである。この範囲外のファイルを作成・変更してはならない。
- {担当ディレクトリ/パッケージ1}
- {担当ディレクトリ/パッケージ2}

■ 触れてはならない範囲
以下は他のエージェントが担当している。絶対に変更するな。
- {他エージェント担当範囲1}
- {他エージェント担当範囲2}

■ 補足・特記事項
- {共通モジュールへの依存がある場合の注意など}

■ 成果物
- {期待する成果物とその格納先}

■ 完了条件
- {何をもって任務完了とするか}
═══════════════════════════════════════
```

### マーカーの記述ルール

| マーカー | 書式 | 例 |
|---------|------|-----|
| `【サブタスク: {id}】` | 工程内で一意なID | `【サブタスク: coding-user】` |
| `【担当範囲: {scope}】` | カンマ区切りのパス | `【担当範囲: src/main/java/com/example/user/, src/main/java/com/example/dto/user/】` |

### 並列発令の手順

**【必須】手順0から始めよ。スキップ禁止。**

0. **設計書の1.3節（ディレクトリ構成）を Read し、全新規ファイルの完全パスを把握する**
   - 設計書にディレクトリ構成が記載されていない場合 → 設計工程に差し戻す
   - 設計書の記載が「ファイル名のみ」で完全パスがない場合 → 設計工程に差し戻す
1. 把握した完全パスをもとに、独立して作業可能なモジュール境界を特定する
2. 共通モジュールがあれば先に1エージェントで完了させる
3. 独立モジュールごとにサブタスク命令書を作成する（【担当範囲】は設計書の完全パスを使用）
4. **1つのメッセージ内で複数の Agent ツールコールを並列発行する**
5. フックがスコープ重複を検知した場合はブロックされるので、分割を見直す

|------|---------------|------|------|
| 要件定義 | 要件定義エージェント | 不可 | 要件の整理・文書化 |

| コーディング | コーディングエージェント | **可** | プログラム実装 |
| ソースレビュー | ソースレビューエージェント | 不可 | コード品質・セキュリティレビュー |
| 単体テスト | 単体テストエージェント | **可** | 単体テスト実施 |
| 強化テスト | 強化テストエージェント | **可** | カバレッジ100%目標テスト |
| 完全テスト | 完全テストエージェント | **可** | 条件網羅テスト実施 |
| 結合テスト | 結合テストエージェント | 不可 | 結合テスト実施 |
| スキル開発 | スキル開発エージェント | 不可 | スキル設計・作成 |
