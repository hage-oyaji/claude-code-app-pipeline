# パイプライン詳細ルール

隊長が参照する詳細ルール。パイプライン開始前にReadすること。

## 開発モード

パイプライン開始時（`/pipeline-start`）に、将軍閣下の指示内容から開発モードを判定し、確認を取ること。

| モード | 値 | 概要 |
|--------|---|------|
| **新規開発** | `new-development` | 既存資産なし。ゼロからシステムを構築する |
| **機能追加** | `feature-addition` | 既存コードベースに機能を追加する |
| **強化テスト** | `enhanced-test` | 既存コードに対してカバレッジ100%目標のテストを実施する |
| **完全テスト** | `complete-test` | 既存コードに対して条件網羅テストを実施する |

### 新規開発モード

事前作業は不要。

```
要件定義 → [承認] → データモデリング → 基本設計 → [承認] → コーディング → ソースレビュー → 単体テスト → 結合テスト → ドキュメントマージ
```

- プロジェクトルール解析: **スキップ**（既存コードが存在しないため）
- 強化テスト・完全テスト: **無効**
- 基本設計は `rules/project-rule.md` を参照しない（存在しないため）

### 機能追加モード

事前条件:
- `artifacts/code/` に既存コードが配置されていること（**必須 — 未配置なら開始不可**）
- `artifacts/data-model/ddl.sql` が存在しない場合 → 将軍閣下にデータモデルを新規作成してよいか確認を取ること

```
要件定義 → [承認] → データモデリング → プロジェクトルール解析 → 基本設計 → [承認] → コーディング → ソースレビュー → 単体テスト → 結合テスト → ドキュメントマージ
```

- 強化テスト・完全テスト: **無効**

### 強化テストモード

事前条件:
- `artifacts/code/` に既存コードが配置されていること（**必須**）
- `rules/project-rule.md` が配置されていること（推奨）

```
強化テスト → 結合テスト → ドキュメントマージ
```

- 要件定義〜単体テスト: **全てスキップ**
- 完全テスト: **無効**
- テスト失敗時、手戻り先の工程がモード外のため **将軍閣下に上申** する

### 完全テストモード

事前条件:
- `artifacts/code/` に既存コードが配置されていること（**必須**）
- `artifacts/data-model/` にDDLが配置されていること（推奨 — SQL条件網羅に必要）
- `rules/project-rule.md` が配置されていること（推奨）

```
完全テスト → 結合テスト → ドキュメントマージ
```

- 要件定義〜単体テスト: **全てスキップ**
- 強化テスト: **無効**
- テスト失敗時、手戻り先の工程がモード外のため **将軍閣下に上申** する

### 各モードの工程有効/無効一覧

| 工程 | 新規開発 | 機能追加 | 強化テスト | 完全テスト |
|------|:-------:|:-------:|:---------:|:---------:|
| 要件定義 | ○ | ○ | - | - |
| データモデリング | ○ | ○ | - | - |
| プロジェクトルール解析 | - | ○ | - | - |
| 基本設計 | ○ | ○ | - | - |
| コーディング | ○ | ○ | - | - |
| ソースレビュー | △ | △ | - | - |
| 単体テスト | ○ | ○ | - | - |
| 強化テスト | - | - | ○ | - |
| 完全テスト | - | - | - | ○ |
| 結合テスト | ○ | ○ | ○ | ○ |
| スキル開発 | △ | △ | - | - |
| ドキュメントマージ | ○ | ○ | ○ | ○ |

○ = 有効、- = スキップ/無効、△ = 隊長判断

**重要: ドキュメントマージは全モード共通の最終工程である。** 結合テスト完了 ≠ パイプライン完了。ドキュメントマージが完了して初めてパイプライン完了となる。隊長判断（△）の工程をスキップする場合も、その工程の status を `skipped` に設定し、次の enabled 工程へ進むこと。

### ソースレビュー工程について

ソースレビューはフック管理対象のパイプライン正式工程である。コーディング工程完了後、単体テスト前に実行される。

- 隊長の判断で実行する（△ = 隊長判断）
- フックによる自動ステータス管理・品質ゲート・リトライ制御の対象
- レビュー結果が `REQUEST_CHANGES` または `REJECT` の場合、隊長がコーディング工程への手戻りを判断する
- レビュー結果が `APPROVE` の場合、次工程（単体テスト）へ進む
- ソースレビューエージェントは**読み取り専用のgitコマンド**（`git diff`, `git show`, `git status`, `git log`）のみ使用可能
- ソースレビュー工程自体は成果物ファイルを出力しない（レビュー結果は報告書としてフックに記録される）

## 工程内並列実行

### 並列実行が可能な工程

| 工程 | 並列可否 |
|------|---------|
| 要件定義 | 不可（1エージェント） |
| データモデリング | 不可（1エージェント） |
| プロジェクトルール解析 | 不可（1エージェント） |
| 基本設計 | 不可（1エージェント） |
| **コーディング** | **可（n エージェント）** |
| ソースレビュー | 不可（1エージェント） |
| **単体テスト** | **可（n エージェント）** |
| **強化テスト** | **可（n エージェント）** |
| **完全テスト** | **可（n エージェント）** |
| 結合テスト | 不可（1エージェント） |
| スキル開発 | 不可（1エージェント） |
| ドキュメントマージ | 不可（1エージェント） |

### 並列起動の前提条件

隊長は以下の条件を**全て**満たした場合のみ、工程内並列を実行してよい:

1. **タスクが完全に分離可能** — 各エージェントの担当範囲（パッケージ、モジュール、ファイル群）が一切重複しない
2. **担当範囲を命令書に明記** — 各命令書に `【サブタスク: {id}】` と `【担当範囲: {scope}】` を記載する
3. **共有リソースの競合がない** — 共通ユーティリティ、共通設定ファイル等を複数エージェントが同時に変更しない

### 担当範囲の分割例

#### コーディング工程

```
基本設計から独立したモジュール単位で分割:
  サブタスク: coding-user     担当範囲: src/main/java/com/example/user/
  サブタスク: coding-product  担当範囲: src/main/java/com/example/product/
  サブタスク: coding-order    担当範囲: src/main/java/com/example/order/

※ 共通モジュール（common/, config/ 等）は1つのサブタスクに集約するか、先に単体で完了させてから並列化する
```

#### 単体テスト工程

```
テスト対象のモジュール単位で分割:
  サブタスク: test-user     担当範囲: src/test/java/com/example/user/
  サブタスク: test-product  担当範囲: src/test/java/com/example/product/
  サブタスク: test-order    担当範囲: src/test/java/com/example/order/
```

### 並列起動の手順

1. 基本設計書を分析し、独立して実装/テスト可能なモジュールを特定する
2. 共有リソース（共通モジュール等）を特定し、競合しない分割計画を策定する
3. **共有リソースがある場合は、まず共通部分を1エージェントで完了させる**
4. `/format-order` でサブタスク命令書を作成する（各命令書に `【サブタスク: {id}】【担当範囲: {scope}】` を必ず記載）
5. 1つのメッセージ内で複数の Agent ツールコールを並列発行する
6. 各エージェントの完了は個別に通知される — 全員の完了で工程完了

### フックによる安全制御

- **スコープ重複検知:** フックが担当範囲の重複を自動検出してブロックする（ディレクトリの包含関係も検知）
- **並列上限:** `pipeline.max_parallel`（デフォルト: 3）を超えてエージェントを起動できない
- **ファイルロック:** pipeline-status.json への並列書き込みはロックファイルで排他制御される
- **並列不許可工程:** coding, unit-test, enhanced-test, complete-test 以外の工程でサブタスクマーカーを使うとフックがブロックする

### サブタスクのエラーハンドリング

- 1つのサブタスクがエラーになっても、他の稼働中サブタスクは継続する
- エラーサブタスクは他のサブタスクの完了後にリトライまたは上申する
- 全サブタスクがエラーまたは完了した時点で、エラーがあれば工程ステータスは `error` になる

## 成果物の引き継ぎ

各工程の inputDirs / outputDir は `pipeline/pipeline-status.json` に定義済み。命令書作成時に参照せよ。
手戻り時は、問題が検出された工程の outputDir も inputDirs に追加し、問題の内容を把握させること。

### コーディング周りの入力最適化

コーディング〜テスト工程では、コンテキスト消費を抑えるため入力を絞っている。

| 工程 | 要件定義 | データモデル | 基本設計 | ルール | コード |
|------|:-------:|:----------:|:-------:|:-----:|:-----:|
| コーディング | × | ○全量 | **今回分のみ** | ○ | **機能追加時のみ** |
| 単体テスト | × | × | × | ○ | ○ |
| 強化テスト | × | × | × | ○ | ○ |
| 完全テスト | × | ○全量 | **今回分のみ** | ○ | ○ |
| 結合テスト | × | × | **今回分のみ** | ○ | ○ |

- **要件定義は読まない** — 要件は基本設計に反映済みのため、二重に渡さない
- **基本設計は今回分のみ** — 隊長が命令書の「参照ドキュメント」で今回の機能に該当するファイルを明示すること
- **データモデルは全量** — テーブル間の整合性確認のため全DDLが必要
- **コードは機能追加時のみ** — 既存コードとの整合性確認のため。隊長が命令書で参照すべきファイルを明示すること。新規開発時は不要

## パイプライン進捗管理

`pipeline/pipeline-status.json` はフック（`.claude/hooks/update-pipeline-status.js`）が自動更新する。隊長が手動で更新する必要はない。

### 工程内並列時の進捗管理

- `pipeline.active_agents[]` で現在稼働中の全エージェントを追跡
- `stages[工程].subtasks[]` で各サブタスクの進捗を管理（id, scope, status, runs）
- フックがファイルロックで並列書き込みの競合を防止
- 全サブタスク完了 → 工程ステータスが自動で `completed` に遷移
- トークン上限の設定・解除は Pipeline Monitor（http://localhost:8089）から行う

### Pipeline Monitor の起動方法

```bash
node tools/pipeline-monitor.js
```

起動後、ブラウザで http://localhost:8089 を開くとダッシュボードが表示される。`pipeline-status.json` の変更を自動検知してリアルタイム更新される。

## 手戻り規則

### 用語定義: リトライと手戻りの違い

| 用語 | 意味 | 上限 | 判断者 |
|------|------|------|--------|
| **リトライ（retry）** | 技術的エラー（タイムアウト、コンテキスト溢れ等）による同一工程の自動再試行 | `retry_policy.max_retries_per_stage`（デフォルト: 2回） | フック自動判定 |
| **手戻り（rework）** | 品質上の問題（テスト不合格、設計不備等）による前工程への差し戻し | 同一工程への手戻り2回超で上申 | 隊長判断 |

この2つは独立した概念であり、カウントも別管理である。混同しないこと。

隊長は**自らの権限で**手戻りを決定・実行する。将軍閣下の承認は不要。

| 検出工程 | 問題の種類 | 手戻り先 |
|----------|-----------|---------|
| データモデリング | 要件の矛盾・不足 | 要件定義 |
| 基本設計 | 要件の矛盾・不足 | 要件定義 |
| 基本設計 | データモデルの不備 | データモデリング |
| プロジェクトルール解析 | 新規開発モードで既存コードが存在しない | スキップ（モード定義に従う） |
| 基本設計 | プロジェクトルールとの不整合 | プロジェクトルール解析 |
| コーディング | 設計の不備・実装不可能な設計 | 基本設計 |
| コーディング | プロジェクトルールとの不整合 | プロジェクトルール解析 |
| ソースレビュー | REQUEST_CHANGES（コード起因） | コーディング |
| ソースレビュー | REQUEST_CHANGES（設計起因） | 基本設計 |
| 単体テスト | テスト不合格（コード起因） | コーディング |
| 単体テスト | テスト不合格（設計起因） | 基本設計 |
| 結合テスト | モジュール間連携の問題（コード起因） | コーディング |
| 結合テスト | モジュール間連携の問題（設計起因） | 基本設計 |
| スキル開発 | スキル仕様の不備（設計起因） | 基本設計 |
| スキル開発 | 既存スキルとの競合 | スキル開発（再実行） |
| ドキュメントマージ | ドキュメント欠損・整合性エラー | 該当する前工程（隊長判断） |
| 強化テスト（テストモード） | テスト不合格（コード起因・設計起因） | **将軍閣下に上申**（開発工程が無効のため） |
| 完全テスト（テストモード） | テスト不合格（コード起因・設計起因） | **将軍閣下に上申**（開発工程が無効のため） |
| 結合テスト（テストモード） | モジュール間連携の問題 | **将軍閣下に上申**（開発工程が無効のため） |

### テストモード（強化テスト・完全テスト）での手戻り

テストモードでは開発工程が無効のため、テスト失敗時に手戻り先の工程を実行できない。
この場合は **将軍閣下に上申** し、コード修正の指示を仰ぐこと。隊長の判断で開発工程を実行してはならない。

**結合テスト工程も同様である。** テストモードでは結合テストの手戻り先（コーディング・基本設計）が無効のため、結合テスト失敗時も将軍閣下に上申せよ。

### 並列実行中の手戻り

- **サブタスクが1つでもエラーの場合:** 他の稼働中サブタスクの完了を待ってから手戻り判断を行う
- **手戻り時はサブタスク状態をリセットし、工程全体を再実行する**
- 手戻り先の工程から順番に再度パイプラインを進めること
- 同一工程への手戻りが **2回** を超えた場合は将軍閣下に上申せよ
- 要件定義への手戻りは影響大のため報告書で必ず言及すること

## エラーリカバリと耐障害性

- フック（`update-pipeline-status.js`）がエージェント失敗を自動検出し、`retry_pending` または `error` ステータスを設定する
- `retry_pending` の場合、`/pipeline-recover` スキルを使用して復旧せよ。**注意:** `retry_pending` 状態では `current_stage` が維持されるため、別工程の開始はフックにブロックされる。同一工程のリトライは可能だが、別工程へ進む場合は `/pipeline-recover` のステータスリセット（手順4）で `current_stage` を解除する必要がある。
- `error`（リトライ上限到達）の場合、将軍閣下に上申せよ
- `suspended`（トークン上限到達）の場合、`/pipeline-recover` スキルの suspended 復旧手順を使用せよ
- パイプラインが中断された場合、`/pipeline-start` のチェックポイント復旧機能で再開せよ

## 品質ゲート

- フックが次工程のエージェント起動時に前工程の成果物を自動検証する（`quality_gates` 定義に基づく）
- フックがブロックした場合、`/quality-check` スキルを使用して対処せよ
- フックの自動検証とは別に、兵の報告書に「問題・懸念事項」「上申事項」がある場合は、次工程に進む前に内容を精査すること
- テスト工程の報告書にテスト失敗がある場合は、手戻り判断を行うこと

## 観測可能性（Observability）

- フックが実行時間・トークン消費量・エラー回数・手戻り回数・サブタスク数を `pipeline-status.json` に自動記録する
- パイプライン完了報告時は `/pipeline-metrics` スキルで観測データを取得し、報告書に含めること
- **注意:** トークン消費量はエージェントのレスポンスから抽出を試みるが、Claude Code の Agent ツールレスポンスにトークン情報が含まれる保証はない。取得できない場合は 0 として記録されるため、トークン値は参考値として扱うこと

## フックのスコープ

フック（`update-pipeline-status.js`）は `settings.json` の `matcher: "Agent"` により全ての Agent ツール呼び出しに反応するが、パイプライン制御の対象となるのは `AGENT_STAGE_MAP` に登録されたエージェント（要件定義エージェント〜スキル開発エージェント）のみである。

- **Explore、Plan 等の非パイプラインエージェント**は `AGENT_STAGE_MAP` に該当しないため、フックはスキップする（ブロックも進捗記録もしない）
- 未知の `subagent_type` が検出された場合、フックは警告ログを出力してスキップする
- 隊長が調査・分析目的で非パイプラインエージェントを起動してもパイプライン状態に影響しない
